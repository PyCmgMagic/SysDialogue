  ▎
  ▎ 2.1 为何采用受限工具面
  ▎
  ▎ AI 与系统之间不直接通过 shell 命令对话，而是通过一组预先定义、参数受约束的 
  ▎ function call 进行交互。这样做有两个目的：一是让每一种系统操作都有稳定、可预
  ▎ 测的调用形态，便于风控层精确识别意图；二是天然排除掉一切未在工具面内声明的能
  ▎ 力，构成一道"能力白名单"。
  ▎
  ▎ 2.2 能力覆盖范围
  ▎
  ▎ 工具面覆盖赛题基础需求所涉及的七类能力：磁盘、文件检索、进程、端口与网络、用
  ▎ 户、服务、系统信息与日志。每一类只暴露完成任务必需的最小能力集，参数经过收敛
  ▎ （例如路径、搜索深度、行数上限），避免出现"一个工具能做无限多件事"的情况。
  ▎
  ▎ 2.3 故意不提供的能力
  ▎
  ▎ 文件写入与删除不纳入工具面。动机是：文件删除与写入的破坏力极高，且与"分析、
  ▎ 诊断、管理"这一赛题核心场景关系较弱，纳入后会显著放大风控复杂度与误操作成本
  ▎ 。当用户提出"清理文件"类需求时，系统仅做分析与建议，让用户在清楚上下文后自行
  ▎ 决定是否执行，从而把不可逆操作的最终决策权保留在人手上。
  ▎
  ▎ 2.4 元工具的必要性

● 这是方案的根基性分歧，不是措辞问题。我先把你的三个点串起来看：

  张力所在：
  1. 纯受限 function call → 覆盖不了长尾，用户需求一旦超出预设 12 个工具就卡住
  2. 参数放宽 function call → 退化成带包装的 shell
  3. 让用户自己执行 shell 兜底 → 违背"去命令行"的赛题方向，也违背"AI 代理"定位
  4. Workflow 预置模板 → 是提前写死的"剧本"，AI
  真要有"代理"能力，不该依赖人工编排模板
  5. Plan 冻结后只能 go 或作废 → 太死板，AI 应能根据执行过程中的新信息动态调整

  我的理解方向（待你确认）：

  - 工具面改为"分层"：精细 function call

  Claude's Rewrite:

  ▎ 模块 5：风险分级
  ▎
  ▎ 风险分级是安全门层的核心组件。每一次工具调用在进入实际执行前都必须先被打上风
  ▎ 险标签，标签决定它走哪一条路径：直接执行、展示后执行、要求用户确认、还是直接
  ▎ 拦截。
  ▎
  ▎ 四级风险标签
  ▎
  ▎ - SAFE：只读、无副作用的操作，直接执行
  ▎ - WARN-LOW：有副作用但可控（如重启非关键服务、大范围磁盘扫描），展示给用户后
  ▎ 执行
  ▎ - 
  ▎ WARN-HIGH：有显著系统影响（如停止关键服务、创建或删除用户、修改用户组、shell
  ▎  通道调用），强制用户二次确认
  ▎ - BLOCK：触达不可接受的操作（如访问内核内存、杀死 init 进程、删除 root 
  ▎ 账号），直接拒绝并向用户解释原因
  ▎
  ▎ 分级的判定维度
  ▎
  ▎ 判定不是按"工具名"一刀切，而是沿着三个维度综合：
  ▎
  ▎ - 操作类型：只读 vs 写 vs 删除 vs 进程终止 vs 服务控制
  ▎ - 目标对象：路径是否触达系统关键文件、进程是否为系统核心进程、服务是否为关键
  ▎ 基础服务、用户是否为超级账号
  ▎ - 上下文：本地模式 vs 远程 SSH 
  ▎ 模式（远程模式下某些操作会引入额外风险，例如停止 sshd 等价于自断连接）
  ▎
  ▎ 同一个工具在不同参数下可能落在不同等级。例如磁盘查询工具默认 
  ▎ SAFE，但递归扫描大目录会升为 WARN-LOW；服务管理工具的 status 动作是 SAFE，但
  ▎  stop/disable 关键服务就会升为 WARN-HIGH。
  ▎
  ▎ 拦截类的防护重点
  ▎
  ▎ BLOCK 层不是一张拍脑袋的黑名单，而是围绕几类"不可恢复"风险组织的：
  ▎
  ▎ - 系统完整性：系统账号库、启动分区、systemd 
  ▎ 单元目录等，一旦破坏系统可能无法启动
  ▎ - 提权通道：sudo 配置、SSH 服务端配置文件等，被篡改会导致安全模型失效
  ▎ - 内核与硬件：内核内存、裸设备等，读写可能直接导致内核崩溃
  ▎ - 反向锁定：远程模式下停止或禁用 SSH 服务——这会当场切断 AI 与目标主机的通道
  ▎ - 进程核心：终止 PID 1、删除 root、修改 root 用户组等
  ▎ - 路径穿越：参数中包含 .. 组件时拒绝，防止 AI 通过相对路径绕过关键路径保护
  ▎ - 资源耗尽：参数范围明显超出合理用途（如全盘深度递归搜索），防止单次调用拖垮
  ▎ 系统
  ▎
  ▎ 所有 BLOCK 
  ▎ 触发都会返回明确的结构化原因，供上层用自然语言向用户解释"为什么不能做"。
  ▎
  ▎ Shell 通道的特殊分级
  ▎
  ▎ shell 通道作为长尾兜底，其风险不可能按参数精确预判——命令文本是自由输入。因此
  ▎ 采取保守策略：
  ▎
  ▎ - 默认 WARN-HIGH：任何 shell 调用都强制用户确认，并完整展示即将执行的命令
  ▎ - 叠加模式匹配：命令文本对照一组已知危险模式（rm -rf 
  ▎ /、写入系统关键路径、管道执行外部脚本等）做前置拦截，命中即升为 BLOCK
  ▎ - 审计完整保留：无论结果如何，命令全文与输出都写入审计日志
  ▎
  ▎ 这样既保留了 shell 通道的兜底能力，又用"强确认 + 模式拦截 + 
  ▎ 完整审计"把风险压到可接受范围。
  ▎
  ▎ 分级优先级
  ▎
  ▎ 当一次调用同时命中多条规则时，按 BLOCK > WARN-HIGH > WARN-LOW > SAFE 取最高
  ▎ 级别。分级器不尝试"降级"——只要有一条规则要求拦截或确认，就必须拦截或确认。

  ▎

  ▎ 程。它的核心角色是把一串工具调用组织成一份用户可审视的计划、引导用户确认、驱
  ▎ 动逐步执行，并允许在执行过程中被动态修改。
  ▎
  ▎ 一份计划是什么
  ▎
  ▎ 一份计划是一个有序的步骤序列。每个步骤包含：要调用的工具、参数、该步骤的用途
  ▎ 说明（用自然语言写给用户看）、以及预期的风险级别。步骤之间可以声明依赖，用来
  ▎ 让下一步使用前一步的结果。
  ▎
  ▎ 计划是结构化的，不是一段自然语言说明。结构化之所以重要，在于"展示给用户看的
  ▎ 计划"与"实际执行的计划"必须完全一致——如果展示时是文字描述、执行时由模型再临
  ▎ 场翻译成工具调用，中间的漂移就会让用户的确认失去意义。
  ▎
  ▎ 从提交到执行
  ▎
  ▎ 用户请求到来、AI 通过元工具提交计划后，引擎做三件事：
  ▎

  ▎ 2. 对每一步预先跑一次风险分级，把预期风险标注在计划上
  ▎ 3. 以结构化形式把计划展示给用户——标出每一步要做什么、风险级别、预期结果
  ▎

  ▎ 认、审计记录、实际执行。一步失败时，引擎停下来等 AI 
  ▎ 做下一步决定（修改计划、跳过、终止）。
  ▎
  ▎ 动态修改的计划
  ▎
  ▎ 计划不是一次提交之后不可改的冻结清单。AI

  ▎ - 新增后续步骤（例如前一步查出的结果表明还需要再清理某个目录）
  ▎ - 删除还未执行的步骤（例如前一步已经解决了问题，后面计划中的步骤失去意义）
  ▎ - 修改还未执行步骤的参数
  ▎

  ▎ 细节。冻结式计划要么把 AI 
  ▎ 限制在"第一版猜测"里，要么每次需要调整就强迫用户推倒重来，两种代价都不合理。

  ▎ 留出响应执行反馈的空间。
  ▎
  ▎ 用户的介入权
  ▎
  ▎ 用户在任意两步之间都可以打断执行，提出修改意见或直接终止。当用户主动介入时，
  ▎ AI 在下一轮对话中会收到完整的执行历史（哪些步骤已完成、当前结果如何），在此
  ▎ 基础上继续规划或调整。这使得"AI 
  ▎ 自主执行"与"人工监督"不是二选一，而是可自由切换的两种节奏。

  ▎ 大模型 API Key 按来源优先级读取：环境变量优先，其次是用户家目录下的配置文件
  ▎ （要求文件权限收紧，避免被其他用户窥视），最后才是主配置文件。一旦读取到来源
  ▎ 较低的配置，启动时会给出明确警告——让用户自己知道"我把密钥留在了风险更高的地
  ▎ 方"，而不是事后才发现。
  ▎
  ▎ 权限检查不是软提示而是硬性拒绝：若配置文件的权限不符合要求，程序直接拒绝读取
  ▎ 并解释原因。动机是把"不安全的密钥存放方式"消灭在启动阶段。
  ▎
  ▎ --verify：自检入口
  ▎
  ▎ 提供一个独立的自检命令，覆盖从基础通路到高阶能力的完整检查：
  ▎
  ▎ - 与大模型 API 的连通性
  ▎ - 基础 OS 工具在当前环境下的可用性
  ▎ - 路径参数注册表的完整性
  ▎ - 风险规则的覆盖率（BLOCK / WARN 规则是否全部可被触发）
  ▎ - 计划模式的动态修改能被正确记录与回放
  ▎ - 远程主机的指纹校验在未知主机与已知主机下都表现正确
  ▎ - 环境探测针对不同发行版都能选中正确的路径
  ▎ - 输出脱敏覆盖所有声明的敏感模式
  ▎ - 故障恢复（超时、断连、权限不足、用户取消）都能回归到稳定状态
  ▎ - 复现包导出的文件完整且可被第三方使用
  ▎
  ▎ 自检存在的动机有两个：新机器拉代码、环境变更后需要一条命令给出明确的"通 / 
  ▎ 不通"答案；同时它也是一份活的"方案完成度"清单——跑不过哪一项，就说明对应功能
  ▎ 未真正落地。
  ▎
  ▎ --demo：演示入口
  ▎
  ▎ 提供一个独立的演示命令，以预设脚本的方式依次展示方案的核心能力：基础四项（磁
  ▎ 盘、进程、端口、用户）、风险拦截、计划模式、动态修改、远程主机指纹确认、环境
  ▎ 自适应等。
  ▎
  ▎ 演示命令的存在动机是为评委 / 观众 / 新用户提供一条零门槛的体验路径。他们不需
  ▎ 要知道"该问什么问题才能触发哪个能力"，直接运行即可看到方案的完整面貌。演示过
  ▎ 程中的所有对话、风险判定、实际执行同样写入审计日志，与正常使用路径保持一致。
  ▎
  ▎ --verify 与 --demo 明确分工：前者回答"能不能用"，后者回答"能用成什么样"。

  ▎ 模块 12：语音输入
  ▎
  ▎ 语音是赛题明确鼓励的多模态交互方向。本模块把语音识别结果纳入对话主流程，让用
  ▎ 户可以用说话代替打字。
  ▎
  ▎ 纯输入层定位
  ▎
  ▎ 语音模块只承担一件事：把用户的语音转成文本，送进与键盘输入完全相同的下游管道
  ▎ 。风险分级、确认机制、审计记录都不因输入方式变化而有任何不同。
  ▎
  ▎ 这样定位的动机是：多模态不应该稀释安全边界。如果语音走一条更"轻"的路径（例如
  ▎ 跳过某些确认），攻击面会立刻扩大——恶意录音、误识别、环境噪音都可能触发未经授
  ▎ 权的操作。把语音做成纯粹的输入通道，上层逻辑对"从键盘来"还是"从麦克风来"保持
  ▎ 不知情，是最简单也最安全的处理。
  ▎
  ▎ 高风险操作不接受语音确认
  ▎
  ▎ 上述"输入方式不影响风险分级"有一个例外：WARN-HIGH 
  ▎ 及以上的确认动作不接受语音作为确认途径。
  ▎
  ▎ 之所以反过来收紧这一条，是因为确认的本质是"用户主动、清晰、可追溯地表达同意"
  ▎ 。语音识别存在误识别与环境干扰的可能性，在破坏力较高的操作上不应当允许"我以
  ▎ 为我说了不要，但系统听成要了"这种场景。确认仍需通过键盘或专用控件完成，语音
  ▎ 可以触发操作，但无法替代用户对操作的最终拍板。
  ▎
  ▎ 识别通路的选择
  ▎
  ▎ 语音识别提供本地与联网两种方案，二者的区别主要在隐私、延迟与准确率的取舍：
  ▎
  ▎ - 本地方案：不依赖网络，语音数据不出本机，隐私最好，但对本机资源有要求
  ▎ - 联网方案：依托云端服务，准确率通常更高，但语音会离开本机
  ▎
  ▎ 用户在配置阶段选定一种，运行时不动态切换。选择结果记录在会话元信息中——若以后
  ▎ 需要分析识别失败案例，能知道当时用的是哪条通路。
  ▎
  ▎ 与对话主流程的接入点
  ▎
  ▎ 语音识别完成后，产出一段文本，连同一个"本条输入来自语音"的标记一起进入对话流
  ▎ 。标记只用于审计与 UI 
  ▎ 展示（例如在消息上标个小图标表示"语音输入"），不参与任何路由或权限判断。

  ▎

  ▎

  ▎ 主张：AI 作为操作系统交互界面，不仅可行，而且可信、可验证。

  第十章：结论

  ▎ 十、结论
  ▎
  ▎ 这份方案的目标不是把功能表铺得更长，而是让"AI 充当操作系统智能代理"这件事在
  ▎ 两个层面都站得住：比赛评审时可被评委独立验证，真实使用时可被用户信任。
  ▎
  ▎ 围绕这个目标，设计里有五条贯穿始终的主线：
  ▎
  ▎ 计划可演进。 
  ▎ 多步任务不是一次提交之后不可改的剧本，而是会随着执行反馈动态调整的状态机。AI
  ▎  能新增步骤、删除无意义的后续、修改还没执行的参数；用户也能在任何两步之间介
  ▎ 入。
  ▎
  ▎ 环境可感知。 AI 的判断不是基于它训练时见过的通用 
  ▎ Linux，而是基于当前这台机器真实的环境画像——发行版、init 
  ▎ 系统、当前用户、命令可用性——这份画像在会话开始时被采集，在关键事件后被刷新。
  ▎
  ▎ 执行可审计。 每一次决策、每一条命令、每一个结果都写进审计日志，沿决策、命令
  ▎ 、结果三条线展开。命令不会出现在用户的对话里，但会完整保留在审计与复现包里，
  ▎ 评委能随时翻到任何一轮看清楚"到底发生了什么"。
  ▎
  ▎ 风险可解释。 分级不是一张拍脑袋的黑名单，而是围绕几类能说清楚的威胁组织的：
  ▎ 系统完整性、提权通道、内核与硬件、远程反向锁定、进程核心、路径穿越、资源耗尽
  ▎ 。每次拦截都能给出结构化原因。
  ▎
  ▎ 长尾可兜底。 精细工具面之外保留一条受控的 shell 通道，由规则 + LLM 两阶段 
  ▎ verifier 把关，默认强制用户确认。这条通道让方案不会因为预设工具面覆盖不全而
  ▎ 在真实场景里中途失能。
  ▎
  ▎ 这五条合在一起想说的其实是同一件事：AI 作为操作系统的交互界面，不仅能跑，而
  ▎ 且可信、可验证、能让人放心地把操作权交给它。
