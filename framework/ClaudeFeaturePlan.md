# 一、整体架构

系统采用六层架构，自上而下依次是：界面层、对话引擎层、AI 调用层、安全门层、执行适配层、工具执行层。

```text
界面层         ──  接收自然语言输入（文本 / 语音）
    ↓
对话引擎层     ──  多轮上下文 / 执行模式路由 / 环境快照注入
    ↓
AI 调用层      ──  与大模型的 agentic loop
    ↓
安全门层       ──  风险分级 / 用户确认 / 审计日志
    ↓
执行适配层     ──  本地 / 远程执行、发行版差异抹平
    ↓
工具执行层     ──  系统能力封装、Shell 通道、DynTool (动态工具)
```

分层的核心动机是把意图理解、风险判定、实际执行彻底解耦。安全门作为独立一层夹在 AI 与执行之间，是整套方案"可信"的根本——无论上层模型输出什么，所有系统副作用都必须在此被分级与审计，这一层不允许任何业务路径绕过。

### 贯穿全局的三条硬约束：

- 对外不暴露 shell 命令：AI 给用户的自然语言回复中不出现 shell 命令字符串，避免用户被诱导去手动执行未经风控的命令，同时呼应赛题"去命令行化"的方向。
- 安全门不可绕过：任何系统副作用必须经过风险分级与必要的用户确认。
- 执行痕迹只进审计：实际执行的命令只写入审计日志与复现包，不回流对话界面——用户看到的是"做了什么、结果如何"，不是"执行了哪条命令"。

# 二、工具面设计

## 2.1 设计原则

- AI 不向用户直接输出 shell 命令字符串，所有系统交互都经由 function call 完成
- 工具面采用分层设计：精细工具覆盖高频场景，DynTool 固化重复能力，shell 通道兜底长尾
- 所有工具调用统一经过安全门，无例外、无绕过路径

## 2.2 能力分层

### 精细 function call

覆盖可穷举、模式稳定的系统操作。将这类能力单独做成结构化工具的好处是：参数可收敛、意图易识别、风险可精准分级。覆盖的能力域包括：

- 磁盘与文件（使用情况、检索、读、写、删）
- 进程与端口
- 用户与服务
- 系统信息、网络信息、日志

### DynTool (动态提案工具)

针对反复出现的能力缺口（如安装特定组件），AI 通过元工具提案创建带参数的命令模板。用户审批一次模板后，固化为精细工具，AI 可多次调用。其价值在于提升高频长尾任务的“去命令行化”体验，避免用户反复确认相同逻辑。

### shell 通道

精细工具无法穷尽所有真实场景。为此保留一个通用 shell 执行工具作为兜底，AI 可向其传入完整命令来处理长尾需求。

引入 shell 通道的动机是让 AI 具备真正意义上的"代理"能力——既不必因工具面受限而中途卡壳，也不必为了覆盖一切把精细工具的参数无限放宽、最终退化成带壳的 shell。

（shell 通道的风控策略，包括命令拦截、确认流程、审计粒度，统一在安全门章节讨论。）

## 2.3 关于文件读写

文件的读、写、删除全部纳入精细工具面，不让用户回退到 shell 手动执行。

理由有两点：一是文件操作是系统管理中最高频的需求之一，强制用户切回命令行与"去命令行化"的方向相悖；二是人机之间频繁的操作交接本身就是出错源头，不如由 AI 在受控链路中一并完成。这些操作的风险分级由安全层统一处理。

## 2.4 元工具

除 OS 工具外，系统向 AI 额外提供一个元工具，用于声明并维护当前任务的执行模式。

### 执行模式仅保留两种：

- 单步直接执行：简单请求直接调用 OS 工具
- 计划模式：多步任务先出计划，用户确认后开始执行

v4.1 中的 workflow 模板模式取消。预置模板本质是"提前写死的剧本"，既与"AI 代理"定位相悖，也无法覆盖真实任务中不断变化的上下文。连续任务统一走计划模式即可。

计划是动态状态机，不是一次性冻结的列表。

AI 首次提交计划、用户确认后进入执行阶段；执行过程中，AI 可根据新获得的信息通过元工具修改计划——新增步骤、删除未执行步骤、调整参数、或提案新的 DynTool 都在允许范围内。每次修改都会写入审计日志，用户也可以在任意步骤间主动介入调整。

元工具本身不产生系统副作用，因此不过风险分级；但每次调用都写入审计，用于还原决策路径。

## 模块 1：AI 调用层

本层负责与大模型之间的 agentic loop：把用户请求、历史上下文、可用工具定义一起交给模型，接收模型返回的 tool call，执行后把结果回传给模型，直到模型认为任务结束。

传递给模型的工具分为两组：OS 工具（含已固化的 DynTool）与元工具（执行模式声明）。模型在每一轮中自主决定调用哪一个。

执行模式按"用户请求"粒度决定。 每一轮用户新输入都是一次独立的模式选择机会，上一轮是单步还是计划不约束下一轮。模式的判定点发生在模型返回的第一个 tool call 上：

- 首个调用是元工具 → 进入计划模式，由计划引擎接管
- 首个调用是 OS 工具 → 按单步模式处理，直接进入安全门

这样做的动机是：让执行模式由模型基于用户请求的复杂度显式表态，而不是由系统用关键词、前缀等启发式方法去猜。模型是最了解"这个请求要几步"的一方，把判断权交给它最合理；同时元工具让模式选择成为一个可审计的结构化事件，而非埋在自然语言里的隐式信号。

单轮内模式不切换。一旦本轮首个 tool call 决定了模式，剩余的执行都保持一致。即使 AI 在执行过程中意识到误判了复杂度，也不在同一轮内升级到计划模式——这样可以避免 UI 状态在执行中途被打断，也让模式选择真正承担"任务开始前的判断"这一角色。若 AI 确实低估了任务规模，由用户在下一轮追问时重新触发计划模式即可。

容错设计： 即使模型在应当规划的场景下跳过了元工具，系统仍能按单步模式继续执行——只是失去计划展示、动态调整与回放能力。元工具是体验增益而非硬性前置，模型偶发的模式误判不会导致功能中断。

（API Key 的读取与校验策略属于配置管理，统一在配置与验证章节讨论。）

## 模块 2：对话管理

本层负责维护多轮对话的上下文，包括历史消息、tool call 结果、当前会话的元信息，以及可供 AI 调用层使用的上下文窗口管理。

按"轮"组织上下文。 每一轮用户输入与 AI 的响应（含所有 tool call 与 tool result）绑定为一个 Turn Bundle。这样组织的好处是：可以按轮裁剪历史、可以按轮回放、也能在触发模型上下文上限时整轮丢弃或压缩早期 bundle，而不会出现"保留了工具调用记录却丢了结果"这种断裂。

### Token 预算管理

- system prompt、工具定义、对话消息、tool result 全部计入预算。针对中文内容做 token 估算修正，避免预算误判
- 每个工具的返回在执行层就做硬截断（见执行章节），防止单次返回爆掉上下文
- 当剩余预算低于阈值时，按 Turn Bundle 为原子单位进行处理

### Compact（对话压缩）

预算压力上升时，最早的 Turn Bundle 不是直接丢弃，而是先交给一个压缩流程，把"用户问了什么、AI 做了什么操作、结果大意如何"浓缩成一段结构化摘要，替换原始 bundle 留在上下文中。

这样做的动机是：原始消息与工具返回体积大但信息密度低，模型真正需要的是"前面做过什么、当前状态如何"的长期线索。压缩后的摘要保留决策连续性，释放出的预算留给后续对话。若压缩后预算仍不足，再按时间顺序整轮丢弃。

### 与审计层的联动

每个 Turn Bundle 关联当前计划 ID（若处于计划模式）与环境画像 ID。任意一轮对话都可通过审计日志精确复现——在哪个环境、基于哪个计划、在什么历史上下文下做出的决策。即使对话层已经把 bundle 压缩或丢弃，原始数据仍保留在审计日志中可供回放。

## 模块 3：系统提示词构建

本层负责在每次调用模型前构造 system prompt，把 AI 的行为边界与当前运行环境一并注入。

### 对外表达 与 对内记录 双轨约束

对用户的自然语言对话中，不输出裸 shell 命令字符串；若用户请求超出工具面覆盖，只描述"需要做什么"的语义，不给出可直接拷贝粘贴的命令行。而在审计层面，完整的命令 trace 照常记录，用于比赛取证与线上复现。

两者并不矛盾：前者是对用户可见层的体验约束，呼应赛题"去命令行化"的方向；后者是对评委、对线上排障的可验证性要求。

### 注入运行时环境快照

system prompt 会动态注入当前运行环境的关键信息，让模型在产出建议时依据真实环境，而非训练时的通用认知。注入内容包括：发行版与版本、本地 / 远程模式、是否容器环境、init 系统类型、当前用户与权限级别、关键命令的可用性矩阵。

这样做的动机是：不同 Linux 发行版之间的工具链差异是赛题的核心考察点之一。把环境画像作为 prompt 的一部分，让模型从一开始就知道自己在什么环境下工作，比事后在工具执行层做 fallback 更干净、也更容易被审计还原。

### Prompt 版本化

每个版本的 system prompt 都会被记录，并与审计日志绑定。线上行为与预期不符时，可精确知道当时使用的 prompt 版本，便于判断是模型问题、prompt 问题，还是环境变化导致。

### 快照的刷新策略

环境画像在会话启动时采集一次，作为基线注入到 system prompt。之后不在每轮对话前重复探测（避免每轮多出探测耗时），但当发生可能改变快照内容的事件时主动刷新相关字段。典型触发场景：执行了服务启停（影响服务可用性）、创建或删除用户（影响当前用户视角）、模式切换（本地 → 远程）等。刷新是字段级而非整体重跑，尽量减少代价。

这样设计的动机是在准确性与成本之间取平衡：纯静态快照可能随着 AI 自身的操作而失真，纯每轮刷新又会把探测成本摊到每一次用户输入上。事件驱动的增量刷新能覆盖绝大多数"画像真的变了"的场景。

## 模块 4：工具注册中心

本层是所有工具（OS 工具、shell 通道、元工具）的唯一注册入口，负责向 AI 调用层提供统一的工具清单，并在系统启动时校验各工具的 schema 完整性。

### 集中注册的动机

把所有工具的定义、参数范围、描述文案收拢到一个地方，带来三个直接收益：

- 单一可信源：避免工具定义散落在各模块中导致 AI 看到的 schema 与实际执行行为漂移
- 便于审计：每次会话开始时可导出"本次 AI 可见的全部工具"，让评委能清楚看到能力边界
- 便于版本化：工具清单与 prompt 版本一起参与版本管理，历史对话的工具能力可被精确复现

### 元工具的职责

元工具在工具注册中心内与 OS 工具并列，承担两类动作：

- 模式声明：用户请求到来时，AI 通过元工具宣告本轮采用单步或计划模式
- 计划修改：计划模式下，AI 在执行过程中通过元工具动态增删或修改计划步骤

元工具的具体调用形态（是一个统一工具通过 action 字段区分，还是拆成多个工具）属于 schema 层面的设计选择。动机上只强调一点：所有的模式声明与计划变更都是结构化、可审计、可回放的事件，绝不通过自然语言 hack 去触发。

### 工具面的可见性分层

注册中心并非把所有工具在任何时刻都暴露给 AI。例如 shell 通道可以默认关闭，仅在用户显式开启"长尾模式"后才加入可见清单；又如文件删除这类高风险工具可以按运行模式（本地/远程）或用户配置做可见性裁剪。这样可以让工具面的"攻击面"随场景收敛。

### 元工具的具体形态：单一工具 + action 字段

元工具在注册中心以一个统一工具（例如 set_execution_mode）的形式暴露，通过 action 字段区分具体动作：声明模式、新增计划步骤、删除未执行步骤、修改步骤参数。

这样设计的动机是让 AI 在"执行模式管理"这一职责上看到单一入口，避免多个语义相近的工具并列导致模型选择混淆。参数的条件性（某些字段只在特定 action 下有效）通过工具描述文案明确表述，交给模型判断。

## 模块 5：风险分级

风险分级是安全门层的核心组件。每一次工具调用在进入实际执行前都必须先被打上风险标签，标签决定它走哪一条路径：直接执行、展示后执行、要求用户确认、还是直接拦截。

### 四级风险标签

- SAFE：只读、无副作用的操作，直接执行
- WARN-LOW：有副作用但可控（如重启非关键服务、大范围磁盘扫描），展示给用户后执行
- WARN-HIGH：有显著系统影响（如停止关键服务、创建或删除用户、修改用户组、shell 通道调用、DynTool 提案审批），强制用户二次确认
- BLOCK：触达不可接受的操作（如访问内核内存、杀死 init 进程、删除 root 账号），直接拒绝并向用户解释原因

### 分级的判定维度

判定不是按"工具名"一刀切，而是沿着三个维度综合：

- 操作类型：只读 vs 写 vs 删除 vs 进程终止 vs 服务控制
- 目标对象：路径是否触达系统关键文件、进程是否为系统核心进程、服务是否为关键基础服务、用户是否为超级账号
- 上下文：本地模式 vs 远程 SSH 模式（远程模式下某些操作会引入额外风险，例如停止 sshd 等价于自断连接）

同一个工具在不同参数下可能落在不同等级。例如磁盘查询工具默认 SAFE，但递归扫描大目录会升为 WARN-LOW；服务管理工具的 status 动作是 SAFE，但 stop/disable 关键服务就会升为 WARN-HIGH。

### 拦截类的防护重点

BLOCK 层不是一张拍脑袋的黑名单，而是围绕几类"不可恢复"风险组织的：

- 系统完整性：系统账号库、启动分区、systemd 单元目录等，一旦破坏系统可能无法启动
- 提权通道：sudo 配置、SSH 服务端配置文件等，被篡改会导致安全模型失效
- 内核与硬件：内核内存、裸设备等，读写可能直接导致内核崩溃
- 反向锁定：远程模式下停止或禁用 SSH 服务——这会当场切断 AI 与目标主机的通道
- 进程核心：终止 PID 1、删除 root、修改 root 用户组等
- 路径穿越：参数中包含 .. 组件时拒绝，防止 AI 通过相对路径绕过关键路径保护
- 资源耗尽：参数范围明显超出合理用途（如全盘深度递归搜索），防止单次调用拖垮系统

所有 BLOCK 触发都会返回明确的结构化原因，供上层用自然语言向用户解释"为什么不能做"。

### Shell 通道与 DynTool 的特殊分级

shell 通道与 DynTool 提案作为长尾兜底与能力扩展，其风险评估采取保守策略：

- 默认 WARN-HIGH：任何 shell 调用或 DynTool 提案都强制用户确认，并完整展示即将执行的命令或模板
- 叠加模式匹配：命令文本对照一组已知危险模式做前置拦截，命中即升为 BLOCK
- 审计完整保留：无论结果如何，命令全文与输出都写入审计日志

这样既保留了系统的灵活性，又用"强确认 + 模式拦截 + 完整审计"把风险压到可接受范围。

### 分级优先级

当一次调用同时命中多条规则时，按 BLOCK > WARN-HIGH > WARN-LOW > SAFE 取最高级别。分级器不尝试"降级"——只要有一条规则要求拦截或确认，就必须拦截或确认。

### Shell 通道与提案的风险判定

shell 通道与 DynTool 提案引入一个独立的命令 verifier：把即将执行的命令或提案模板交给一个专门的判定流程做语义级评估，输出本次调用的建议风险级别与理由。

verifier 的输出遵循两条硬约束：

- 下限是 WARN-LOW：即使 verifier 判断内容非常安全，也不允许降到 SAFE。保留最低限度的用户感知（展示详情 + 快速放行）能确保用户永远知道系统动用了长尾或扩展能力，不会被无感绕过
- 不确定即升级：verifier 给出不确定、失败或理由不充分的结果时，默认升为 WARN-HIGH 或 BLOCK，不给模糊地带留空间

verifier 的判定结果与理由一并写入审计日志。用户在确认界面能看到判定理由；事后也能回溯 verifier 本身的行为是否稳定。

### 为何用 verifier 而非规则匹配

纯正则模式匹配对 obfuscated 命令非常脆弱；同时又容易误伤合法命令。一个能理解语义的 verifier 更接近人类审查员的视角。同时，将 verifier 视为"也可能出错的组件"，用"不可降到 SAFE"的硬下限作为兜底，即使 verifier 被骗，用户感知层也还在。

### 两阶段 verifier

verifier 采用规则 + LLM 的两阶段设计：

- 第一阶段：硬规则快速过滤。对一批明显不可接受的命令模式做前置拦截，命中即判 BLOCK，不进入 LLM
- 第二阶段：LLM 语义判定。规则未命中的内容交给一次轻量 LLM 调用，送入文本与必要的上下文（当前用户、工作目录、运行模式等），结构化返回建议风险等级与理由

这样分工的动机：大量显而易见的危险命令应当在最便宜的层面被拦住，避免让 LLM 为琐碎判断付出延迟与成本；同时规则层能保证"至少这些必然危险"不会因为 LLM 偶发的错误判断而漏掉。剩下真正需要语义理解的长尾交给 LLM，最大化精度。

## 模块 6：审计日志

审计日志在本方案中不是一份"出了问题才翻"的运行日志，而是整套系统对外证明自己可信的主要手段。它同时服务三类场景：事后追查安全问题、向评委证明 AI 的行为真实可验证、以及把一次对话完整"倒带"进行线上复现。

### 沿三条线记录

每条审计记录不是把"做了什么"塞进一行完事，而是沿三条线平行展开：

- 决策线：AI 当时看到了哪些上下文、风险分级的结果是什么、verifier 给出了怎样的判断、用户有没有确认，最终为什么执行或为什么拦截
- 命令线：实际落到执行层的命令与参数，而不是对它的自然语言描述
- 结果线：执行耗时、退出码、截断后的输出预览、是否触发了环境画像的事件刷新

之所以这样拆分，是因为赛题要求作品能讲清楚"意图理解、实际执行、最终反馈"三个层面。一条扁平记录很难同时回答三种提问角度，分线记录后每条线相互独立又彼此关联，评委从任一角度切入都能顺藤摸到其他两条。

### 对话界面与审计界面的分工

命令线只流向审计面板与复现包，不会回到用户的对话界面。用户看到的是自然语言反馈——做了什么、结果怎样；评委则可以通过审计面板看到底层完整的命令。同一次执行，两套视图，各自服务不同的受众。

这样处理既符合"去命令行化"的赛题方向——命令行不会从后门悄悄回到用户面前，也没有牺牲掉比赛需要的取证深度。

### 元信息与复现

每条记录都带上足以重建上下文的元信息：所在会话、所在计划（如有）、所用的 prompt 版本、所在环境画像 ID、当前的执行模式。这些字段任何一项缺失，复现都会断链，所以记录格式做强校验。

### 两种导出形式

- 审计日志导出：按会话导出完整的结构化日志，适合评委逐条核对
- 复现包导出：在审计日志之外，额外打包环境画像、prompt 版本、演示脚本、截图索引，生成一个可以独立分发的包。第三方拿到之后不需要接触原始机器，也能搞清楚"这次演示到底做了什么、凭什么相信它真的发生过"

复现包是方案"可真实验证"这一主张的最终落点，它的完整程度会直接影响评委对可验证性的评分。

## 模块 7：环境探测与执行适配

这一层夹在安全门和具体工具之间，把一条已批准的调用真正下达到目标系统。它需要解决三件事：搞清楚目标环境长什么样、把命令送到该去的地方、在执行路径不可用时有兜底。

### 环境探测

会话启动时采集一份能力画像：发行版、init 系统、当前用户与权限、是否容器，以及一组关键命令（systemctl、service、journalctl、ss、netstat、ip、ifconfig、sudo 等）的可用性。这份画像既注入到 system prompt，也在下面的路径选择里直接使用。

底层之所以也需要环境感知，是因为 openEuler、CentOS、Ubuntu Server 上做同一件事的命令并不一定相同。把差异集中到这一层处理，上层工具就只关心"做什么"，不必纠结"在这台机器上该用哪个命令"。

### 优先路径 + fallback

每个能力域都绑定一组优先路径和兜底路径：

- 服务管理：systemctl → service
- 日志读取：journalctl → /var/log/*
- 端口查询：ss → netstat
- 网卡信息：ip → ifconfig

选择在执行前依据画像一次性完成，不每次调用都重新探测。

### 本地与远程共用一套接口

本地执行器和 SSH 远程执行器向上层暴露同样的契约：传入命令参数，拿到输出与退出码。上层工具的代码在两种模式下完全一致，审计日志的格式也对齐。

远程连接采用"每条命令独立 SSH exec"而不是持久 shell 会话。这样每次执行都是原子的：前一条命令不会通过环境变量或工作目录状态影响后一条，也让每次调用在审计里都是一条独立、可追溯的记录。

### 远程主机信任

远程模式默认开启严格校验，只接受 known_hosts 里已有的主机指纹。遇到未知主机时不自动信任，而是取回 SHA256 指纹展示给用户，用户显式确认后再写入本地 known_hosts，然后以严格模式重新连接。

默认信任未知主机等于给中间人留门。赛题也强调"环境感知与安全可解释"，所以首次连接的信任必须由用户明确做出，而不是被省略掉。

### 横切关切的统一封装

本地和远程共享一层 SafeExecutor，集中处理跨执行器都要面对的问题：超时、输出截断、退出码归一化、权限不足、SSH 断连与重试。上层工具因此只需要解读结果，不必每个工具都重写一遍容错逻辑。

对话管理章节提到的"tool result 硬截断"——防止单次返回撑爆上下文的那道防线——也落在这里。

## 模块 8：工具实现与输出脱敏

工具层在参数校验通过、风险分级放行之后，真正调用执行适配层完成工作。各工具之间的实现彼此独立，但共享两条公共约束：参数在工具内部做最后一次合法性检查，输出在返回上层之前统一过一道脱敏。

### 工具的参数约束

即使 AI 已经通过工具 schema 给出合法的调用，工具内部仍做最后一次收敛，例如：

- 日志工具把 lines 限制在合理上下限，避免 AI 传入百万行请求
- 文件检索工具把搜索深度、结果条数做硬上限
- 文件读写工具在实际 IO 前再校验路径是否仍在允许范围内

这层防御之所以必要，是因为 schema 校验只能保证类型合法，不能保证语义合理。工具侧多一道收敛，能防止极端参数把系统资源打满，也能兜住 schema 定义里可能漏掉的约束。

### 输出脱敏（OutputSanitizer）

所有工具的输出在交付到上层（对话界面、审计日志、复现包）之前，都先统一过一遍脱敏层。脱敏覆盖的典型敏感信息包括：

- 认证相关：Bearer token、Authorization 头、各类密钥前缀（sk-、AK/SK 等）
- 凭据传递：password=、--password 参数、交互式输入的口令残留
- 连接字符串：数据库连接串、带认证信息的 URL

脱敏采用"保留语义、抹掉原值"的做法——例如把 token 值替换成占位符，但保留字段名与大致长度，便于用户理解"这里有一个 token"而不泄露 token 本身。

### 统一出口

脱敏不是每个工具各自实现，而是放在工具输出的公共出口。这样做的动机是：

- 一处修改，全局生效——新增一类敏感模式只需更新一处
- 对话界面、审计日志、复现包看到的是同一份脱敏后内容，不会出现"UI 看到是脱敏的、日志里却是明文"这种不一致
- shell 通道的自由输出也在同一道防线内，不会因为"这是 shell 执行的结果"就绕过脱敏

这一层保护的直接受益者是审计与复现包——它们本应作为对外可分发的材料，一旦混入真实 token、密码，整份材料就无法拿出示人。脱敏是复现包能够公开展示的前提。

## 模块 9：计划引擎

当 AI 判定当前请求需要多步操作、通过元工具进入计划模式后，计划引擎接管后续流程。它的核心角色是把一串工具调用组织成一份用户可审视的计划、引导用户确认、驱动逐步执行，并允许在执行过程中被动态修改。

### 一份计划是什么

一份计划是一个有序的步骤序列。每个步骤包含：要调用的工具、参数、该步骤的用途说明（用自然语言写给用户看）、以及预期的风险级别。步骤之间可以声明依赖，用来让下一步使用前一步的结果。

计划是结构化的，不是一段自然语言说明。结构化之所以重要，在于"展示给用户看的计划"与"实际执行的计划"必须完全一致——如果展示时是文字描述、执行时由模型再临场翻译成工具调用，中间的漂移就会让用户的确认失去意义。

### 从提交到执行

用户请求到来、AI 通过元工具提交计划后，引擎做三件事：

1. 校验计划里的每一步都是合法的工具调用（工具名存在、参数符合 schema）
2. 对每一步预先跑一次风险分级，把预期风险标注在计划上
3. 以结构化形式把计划展示给用户——标出每一步要做什么、风险级别、预期结果

用户确认后，引擎按顺序驱动执行。每一步走完整的安全门流程：风险分级、必要时确认、审计记录、实际执行。一步失败时，引擎停下来等 AI 做下一步决定（修改计划、跳过、终止）。

### 动态修改的计划

计划不是一次提交之后不可改的冻结清单。AI 在执行过程中可以通过元工具对计划做调整：

- 新增后续步骤（例如前一步查出的结果表明还需要再清理某个目录）
- 删除还未执行的步骤（例如前一步已经解决了问题，后面计划中的步骤失去意义）
- 修改还未执行步骤的参数

每次修改都作为一个结构化事件写入审计日志，也在 UI 上同步通知用户。已经执行完的步骤不可修改——它们是既成事实，只能通过新增后续步骤来纠正。

之所以让计划可动态修改，是因为真实的系统管理任务几乎不可能在开始时就看清全部细节。冻结式计划要么把 AI 限制在"第一版猜测"里，要么每次需要调整就强迫用户推倒重来，两种代价都不合理。把计划做成可演进的状态机，在保留"用户知情同意"前提下给 AI 留出响应执行反馈的空间。

### 用户的介入权

用户在任意两步之间都可以打断执行，提出修改意见或直接终止。当用户主动介入时， AI 在下一轮对话中会收到完整的执行历史（哪些步骤已完成、当前结果如何），在此基础上继续规划或调整。这使得"AI 自主执行"与"人工监督"不是二选一，而是可自由切换的两种节奏。

### 对修改步骤的确认策略

新增或修改的步骤在执行前是否需要用户重新确认，采取用户可配置的方案：

- 默认策略（按风险级别判断）：SAFE / WARN-LOW 的新增步骤自动执行，仅在 UI 上通知；WARN-HIGH 及以上仍然要求用户确认。与单步模式的风险处理对齐，避免频繁打断用户
- 严格模式（可在会话启动时开启）：任何对计划的修改都需要用户点头，即使是 SAFE 级的新步骤也不例外。适合生产环境或用户希望完全掌握节奏的场景

默认策略优先保障流畅度，严格模式优先保障控制感。配置写入会话元信息，审计日志也会记录本次会话采用的策略，便于事后核查。

## 模块 10：配置、自检与会话管理

本层承担三件事：加载与校验运行时配置、提供自检与演示入口、管理会话的存续。

### API Key 的加载

大模型 API Key 按来源优先级读取：环境变量优先，其次是用户家目录下的配置文件（要求文件权限收紧，避免被其他用户窥视），最后才是主配置文件。一旦读取到来源较低的配置，启动时会给出明确警告——让用户自己知道"我把密钥留在了风险更高的地方"，而不是事后才发现。

权限检查不是软提示而是硬性拒绝：若配置文件的权限不符合要求，程序直接拒绝读取并解释原因。动机是把"不安全的密钥存放方式"消灭在启动阶段。

### --verify：自检入口

提供一个独立的自检命令，覆盖从基础通路到高阶能力的完整检查：

- 与大模型 API 的连通性
- 基础 OS 工具在当前环境下的可用性
- 路径参数注册表的完整性
- 风险规则的覆盖率（BLOCK / WARN 规则是否全部可被触发）
- 计划模式的动态修改能被正确记录与回放
- 远程主机的指纹校验在未知主机与已知主机下都表现正确
- 环境探测针对不同发行版都能选中正确的路径
- 输出脱敏覆盖所有声明的敏感模式
- 故障恢复（超时、断连、权限不足、用户取消）都能回归到稳定状态
- 复现包导出的文件完整且可被第三方使用

自检存在的动机有两个：新机器拉代码、环境变更后需要一条命令给出明确的"通 / 不通"答案；同时它也是一份活的"方案完成度"清单——跑不过哪一项，就说明对应功能未真正落地。

### --demo：演示入口

提供一个独立的演示命令，以预设脚本的方式依次展示方案的核心能力：基础四项（磁盘、进程、端口、用户）、风险拦截、计划模式、动态修改、远程主机指纹确认、环境自适应等。

演示命令的存在动机是为评委 / 观众 / 新用户提供一条零门槛的体验路径。他们不需要知道"该问什么问题才能触发哪个能力"，直接运行即可看到方案的完整面貌。演示过程中的所有对话、风险判定、实际执行同样写入审计日志，与正常使用路径保持一致。

--verify 与 --demo 明确分工：前者回答"能不能用"，后者回答"能用成什么样"。

### 会话管理

会话以目录 + 审计文件的形式落盘，保留一段时间后自动清理（例如 30 天），同时限制并存会话数（例如 20 个），防止长期运行下会话目录无限膨胀。用户可以列出当前所有会话、删除指定会话、或基于某个会话 ID 导出审计日志与复现包。

会话的持久化是复现能力的前提——赛题强调作品需要可复现验证，一个结束即消失的系统无法提供这种保证。保留期和上限则是对持久化代价的合理约束。

## 模块 11：终端界面

作为主要的用户界面，终端界面承担输入、输出、状态展示与确认交互四类职责。选择 TUI 而非 Web / GUI 是一个刻意的决策：服务器运维的真实场景大量发生在远程登录的终端中，TUI 能直接嵌入用户既有的工作流，不要求额外的图形环境，也避免了前后端分离带来的部署与数据传输开销。

### 核心界面构成

- 对话主区：按轮组织的消息块，一次用户输入与其对应的 AI 响应在视觉上绑定在一起，便于用户追溯上下文
- 状态面板：持续展示当前会话的关键信息，包括连接模式（本地 / 远程）、远端主机标识、当前执行模式（单步 / 计划中）、预算占用等
- 命令输入：接收自然语言输入，支持多行与编辑
- 确认弹窗：在触发 WARN-HIGH 或 BLOCK 的节点上中断流程，展示风险说明与即将执行的操作，等待用户显式回应

### 审计面板

一个独立的切换视图（通过快捷键进入），展示当前会话的审计日志，重点呈现决策线与命令线：对每次工具调用，用户能看到 AI 当时的风险判定、verifier 的结论、用户是否确认、最终执行了什么命令、结果如何。

审计面板在对话主区之外单开的动机是：命令细节不适合回流到用户日常对话视图（与"去命令行化"冲突），但评委、运维用户、调试者又需要一条随时可用的路径查看"底层到底发生了什么"。两个视图共用同一数据源，只是信息密度不同。

### 环境画像面板

另一个独立视图，展示当前会话所使用的环境画像：目标主机的发行版、init 系统、当前用户、关键命令可用性等。

这个面板存在的意义是让"环境感知"从一个不可见的系统内部状态变成用户可直接核验的信息——当 AI 给出的建议看起来奇怪时，用户可以翻到这里确认"它是不是在用错误的环境假设做判断"。

### 远程首次连接的指纹确认

当远程模式遇到未记录的主机时，界面在任何其他动作之前先弹出一个专门的指纹确认界面：显示主机的 SHA256 指纹、说明当前处于"未信任"状态、解释用户同意后将发生什么。用户显式同意后才写入 known_hosts 并继续；拒绝则直接终止连接。

把指纹确认做成专门的界面、而不是一行警告后默默放行，动机是让"是否信任这台主机"成为一个用户无法忽略的决策——这是前面执行适配层已经强调过的安全原则，在 UI 上必须以相同强度表达。

## 模块 12：语音输入

语音是赛题明确鼓励的多模态交互方向。本模块把语音识别结果纳入对话主流程，让用户可以用说话代替打字。

### 纯输入层定位

语音模块只承担一件事：把用户的语音转成文本，送进与键盘输入完全相同的下游管道。风险分级、确认机制、审计记录都不因输入方式变化而有任何不同。

这样定位的动机是：多模态不应该稀释安全边界。如果语音走一条更"轻"的路径（例如跳过某些确认），攻击面会立刻扩大——恶意录音、误识别、环境噪音都可能触发未经授权的操作。把语音做成纯粹的输入通道，上层逻辑对"从键盘来"还是"从麦克风来"保持不知情，是最简单也最安全的处理。

### 高风险操作不接受语音确认

上述"输入方式不影响风险分级"有一个例外：WARN-HIGH 及以上的确认动作不接受语音作为确认途径。

之所以反过来收紧这一条，是因为确认的本质是"用户主动、清晰、可追溯地表达同意"。语音识别存在误识别与环境干扰的可能性，在破坏力较高的操作上不应当允许"我以为我说了不要，但系统听成要了"这种场景。确认仍需通过键盘或专用控件完成，语音可以触发操作，但无法替代用户对操作的最终拍板。

### 识别通路的选择

语音识别提供本地与联网两种方案，二者的区别主要在隐私、延迟与准确率的取舍：

- 本地方案：不依赖网络，语音数据不出本机，隐私最好，但对本机资源有要求
- 联网方案：依托云端服务，准确率通常更高，但语音会离开本机

用户在配置阶段选定一种，运行时不动态切换。选择结果记录在会话元信息中——若以后需要分析识别失败案例，能知道当时用的是哪条通路。

### 与对话主流程的接入点

语音识别完成后，产出一段文本，连同一个"本条输入来自语音"的标记一起进入对话流。标记只用于审计与 UI 展示（例如在消息上标个小图标表示"语音输入"），不参与任何路由或权限判断。

# 四、模块间数据流

为了说明各模块如何协作，以一个典型的计划模式场景为例：

用户输入："帮我查一下哪些进程占 CPU 最多，把超过 80% 的进程停了"

### 流程：

1. 语音或终端输入进入界面层，转成文本后交给对话管理
2. 对话管理把当前轮消息与历史上下文、system prompt（已注入环境画像）、工具清单一同交给 AI 调用层
3. AI 判断这是一个多步任务（先看进程，再决定杀谁），首个 tool call 调用元工具声明进入计划模式，并提交初始计划：先列进程 → 展示结果 → 根据结果决定杀哪些
4. 计划引擎校验步骤合法性、为每步预跑风险分级，把结构化计划展示给用户
5. 用户在 UI 上确认计划
6. 引擎驱动第一步：列进程。该调用经安全门（SAFE，直接放行）、执行适配层（依据环境画像选择 ps 参数形态）、工具层执行、输出经脱敏，返回 AI
7. AI 看到结果后判断需要新增步骤：对两个超过 80% 的进程分别调用 kill。AI 通过元工具向计划追加两个新步骤
8. 根据用户配置的修改确认策略：kill 属 WARN-HIGH，弹出确认。用户确认后执行
9. 每一步的决策、命令、结果都写入审计日志，附带 plan_id 与 env_profile_id
10. 任务结束后， AI 以自然语言向用户总结"做了什么、结果如何"——不包含裸命令

这张数据流的作用：让读者看清每一次真实用户请求在系统里流经哪些关键节点、在哪里被拦截、在哪里被记录、在哪里产生用户可见的输出。各模块之间不是松散的功能集合，而是围绕"请求 → 决策 → 风控 → 执行 → 记录 → 反馈"的一条清晰链路。

# 五、关键设计决策总结

本章把散落在各模块中的设计决策汇总成一张表，便于快速回顾"方案的几个关键取舍是什么、为什么这样选"。新读者可以从这张表切入，再回到相关模块看细节。

| 决策 | 选择 | 核心原因 |
|---|---|---|
| AI 的对外表达形式 | 自然语言反馈，不输出裸 shell 命令 | 呼应赛题"去命令行化"，避免用户被诱导手动执行未经风控的命令 |
| 执行轨迹如何保留 | 完整命令只进入审计与复现包，不回流对话 | 同时满足"用户体验去命令行化"与"评委可核对实际执行"两个目标 |
| 长尾能力如何兜底 | 保留受控 shell 通道作为兜底，默认最高风险 | 避免工具面受限导致 AI 无法处理真实场景 |
| Shell 通道 vs DynTool | Shell 处理一次性命令；DynTool 处理可参数化的复用能力 | 平衡“万能兜底”的安全成本与“重复操作”的交互效率 |
| Shell 命令的风险判定 | 规则 + LLM verifier 两阶段，下限为 WARN-LOW | 纯规则脆弱、纯 LLM 有成本，两阶段折中；下限确保 shell 使用始终被用户感知 |
| 计划的形态 | 结构化、可动态修改的状态机，而非一次冻结 | 真实任务无法在起步时看全，冻结式计划要么限制 AI 要么反复打断用户 |
| 工作流预置模板 | 整体去掉，连续任务统一走计划模式 | 预置模板是"写死的剧本"，与"AI 代理"定位相悖 |
| 执行环境的差异处理 | 会话启动一次性采集环境画像，事件驱动增量刷新 | 在准确性与探测成本之间取平衡 |
| 远程主机信任 | 默认严格校验，未知主机需用户显式确认指纹 | 避免对中间人攻击开门，同时作为"安全可解释"的展示点 |
| 敏感信息处理 | UI、日志、导出包共用同一道脱敏出口 | 一处修改全局生效，避免视图间不一致；复现包能安全对外分发的前提 |
| 自检与演示 | 区分 --verify 与 --demo 两个独立入口 | 前者回答"能不能用"，后者回答"能用成什么样"，受众不同 |

# 六、实现优先级

功能文档不强规定实现顺序，但列出一个建议的分层优先级，便于团队在资源受限时判断哪些必须先出、哪些可以后补。

### P0 · 主干能力

没有这些方案就跑不通：AI 调用层、工具注册中心、风险分级与安全门、环境探测、本地执行器、审计日志（含决策线与命令线）、终端基本界面、配置加载与启动就绪检查。

### P1 · 完整功能面

使方案达到赛题要求的完整体验：计划引擎（含动态修改）、元工具、shell 通道与两阶段 verifier、远程执行器与主机信任流程、所有 OS 工具（磁盘 / 文件 / 进程 / 端口 / 用户 / 服务 / 日志）、输出脱敏、审计面板与环境画像面板、对话压缩、--verify 与 --demo 入口。

### P2 · 加分项

提升方案的完成度与说服力：故障注入与稳定性矩阵、复现包导出、性能指标面板。

### P3 · 可选增强

提升交互广度，不影响核心评分：语音输入、会话管理面板、rollback 建议等。

### 优先级的含义

优先级描述的是"必要性递减"，不是"质量递减"。任何一层被做到能用即可算达成，做到精细则是加分——例如 P0 的终端界面只要能对话、能确认就算过关，但做成一个可读性强、状态切换流畅 of TUI 会对用户体验与工程质量评分有直接收益。

# 第八章：提交物清单

## 八、提交物

提交物分四类，对应赛题对"代码、Agent、验证材料、合规"的四个角度。

工程与源代码 是让评委能把方案自己跑起来的基础。完整源代码之外，还需要一份 README、依赖清单、示例配置，以及一段环境搭建说明。README 要同时照顾两类读者：想跑通的人关心怎么启动，想审阅的人关心为什么这样组织。

Agent 相关材料 是赛题专门针对"以 Prompt 和工具编排为主"的方案要求的内容。包括 Agent 的整体配置说明、核心 Prompt、工具与能力定义，以及一份"决策边界说明"——讲清楚什么请求走单步、什么请求进计划模式、什么请求直接被拒绝。这份材料要让不看代码的评委也能理解 AI 的行为边界。

验证与复现材料 是"方案可真实验证"这个主张的落点。包括按会话导出的审计日志、完整的复现包（里面打包了环境画像、prompt 版本、演示脚本、截图索引）以及演示视频。这几项任意一项缺失，评委就很难独立判断方案到底有没有跑起来。

合规材料 不决定评分高低，但缺了会直接出合规问题：第三方开源组件清单、各组件的许可证说明、外部参考资料的引用标注。属于必须齐备的一档。

# 第十章：结论

## 十、结论

这份方案的目标不是把功能表铺得更长，而是让"AI 充当操作系统智能代理"这件事在两个层面都站得住：比赛评审时可被评委独立验证，真实使用时可被用户信任。

围绕这个目标，设计里有五条贯穿始终的主线：

计划可演进。 多步任务不是一次提交之后不可改的剧本，而是会随着执行反馈动态调整的状态机。AI 能新增步骤、删除无意义的后续、修改还没执行的参数；用户也能在任何两步之间介入。

环境可感知。AI 的判断不是基于它训练时见过的通用 Linux，而是基于当前这台机器真实的环境画像——发行版、init 系统、当前用户、命令可用性——这份画像在会话开始时被采集，在关键事件后被刷新。

执行可审计。 每一次决策、每一条命令、每一个结果都写进审计日志，沿决策、命令、结果三条线展开。命令不会出现在用户的对话里，但会完整保留在审计与复现包里，评委能随时翻到任何一轮看清楚"到底发生了什么"。

风险可解释。 分级不是一张拍脑袋的黑名单，而是围绕几类能说清楚的威胁组织的：系统完整性、提权通道、内核与硬件、远程反向锁定、进程核心、路径穿越、资源耗尽。每次拦截都能给出结构化原因。

长尾与扩展可控。 精细工具面之外保留受控的 shell 通道与可提案的 DynTool，由规则 + LLM 两阶段 verifier 把关，默认强制用户确认。这条通道让方案不会因为预设工具面覆盖不全而在真实场景里中途失能。

这五条合在一起想说的其实是同一件事：AI 作为操作系统的交互界面，不仅能跑，而且可信、可验证、能让人放心地把操作权交给它。
